# 错题推荐算法 V2 修改方案

## 目标

- 保留现有数据结构和状态逻辑
- 保留 streak、wrong_count、last_review、next_review
- 不引入新的状态分类（Learning/Review/Relearning）
- 利用已有 created_at 和 review_count 字段
- 解决新题不容易被推荐的问题
- 引入轻量级遗忘曲线思想

## 二、核心公式变更

### 旧公式

```
score = (1.0 + overdue_score) x last_wrong_bonus(5) x new_question_bonus(2) x error_rate_bonus x randomness(0.9~1.1)
```

### 新公式

```
score = (1.0 + forget_risk) x freshness_bonus x last_wrong_bonus(3) x error_rate_bonus x randomness(0.95~1.05)
```

## 三、逐项修改详情

### 修改项 1：删除 `overdue_score`

**位置**: `recommendation.rs:178-189`

删除以下代码块及其从未复习时默认给 10.0 的逻辑。

**原因**: 随时间无限线性增长（60 天 = 600 分），导致旧题垄断推荐。

---

### 修改项 2：新增 `forget_risk`（软对数上限）

**位置**: `recommendation.rs`，`calculate_score()` 函数

**公式**:

```
expected_interval = ((due_at - last_review_at) / DAY_SECONDS) . max(1.0)
days_passed        = ((now - last_review_at) / DAY_SECONDS) . max(0.0)
forget_risk        = log2(days_passed / expected_interval + 1)
```

**边界处理**:

- 从未复习题（`last_review_at` 或 `due_at` 为 None）: `forget_risk = 0`
- `due_at == last_review_at`: `expected_interval` 至少为 1.0，避免除零
- `due_at < last_review_at`（历史数据异常）: `expected_interval` 至少为 1.0，避免负数

> 原因：新题尚未进入记忆过程，不应被视为高遗忘风险。新题推荐权重应由 freshness_bonus 提供，而不是 forget_risk。同时避免了 `(1+5) x 5 = 30` 倍的过高新题加权。

**效果对比**:

| 超期倍数 | 硬上限 `min(x,5)` | 软上限 `log2(x+1)` |
|---------|--------------------|---------------------|
| 0 倍（刚复习） | 0 | 0 |
| 1 倍（到期当天） | 1 | 1.0 |
| 2 倍 | 2 | 1.58 |
| 3 倍 | 3 | 2.0 |
| 4 倍（原封顶） | 4 | 2.32 |
| 7 倍 | 5 | 3.0 |
| 15 倍 | 5 | 4.0 |
| 31 倍 | 5 | 5.0 |
| 63 倍 | 5 | 6.0 |

**优势**: 没有突变点，增长平滑自然。极度超期的题目不再需要额外封顶处理。

**评分中的贡献**: `1 + forget_risk` 范围从 1.0（刚复习）到约 7.0（极端超期），远比旧方案 `1 + 600 = 601` 可控。

---

### 修改项 3：新增 `freshness_bonus`

**位置**: `recommendation.rs`，`calculate_score()` 函数

**公式**（仅在 `review_count == 0` 时启用）:

```
days_since_created = (now - created_at) / DAY_SECONDS . max(0.0)
freshness_bonus    = (5.0 - days_since_created x 0.25) . max(1.0)
```

| 创建天数 | bonus |
|----------|-------|
| 0 天 | 5.0 |
| 4 天 | 4.0 |
| 8 天 | 3.0 |
| 12 天 | 2.0 |
| 16 天+ | 1.0 |

> 衰减系数取 0.25 而非原方案的 0.5，原因是 0.5 约 8 天后降至最低值，对于不一定每天使用的错题本来说衰减过快。

**注意**: `review_count` 需要从数据层获取。当前 `calculate_score()` 接收的是 `Question` 对象，而 `Question` 结构体中没有 `review_count` 字段。需要在 `enrich_error_rates` 阶段一并购入 `review_count`（见修改项 7），然后将 `review_count` 传入评分过程。

---

### 修改项 4：`error_rate_bonus` 平滑处理

**位置**: `recommendation.rs`，`enrich_error_rates()` 函数（约第 236-241 行）

**旧公式**:

```
1.0 + error_rate x 2.0
```

**新公式**:

```
1.0 + error_rate x log2(review_count + 1)
```

其中 `review_count` 来自 `review_summary` 视图。

**效果对比**:

| review_count | error_rate | 旧 bonus | 新 bonus | 说明 |
|-------------|-----------|---------|---------|------|
| 1 | 100% | 3.0 | 2.0 | 单次错误不被过度放大 |
| 3 | 66% | 2.32 | 2.32 | 与旧方案持平 |
| 5 | 40% | 1.80 | 1.83 | 持平附近 |
| 10 | 30% | 1.60 | 1.88 | 更多样本，信任度更高 |
| 20 | 25% | 1.50 | 2.0 | 确认真难，权重提高 |
| 50 | 20% | 1.40 | 2.13 | 稳定的高错误率获得充分加权 |

**核心思想**: `review_count` 低时，error_rate 不可靠（可能只是一两次意外），权重低；`review_count` 高时，error_rate 可信度高，权重相应放大。

---

### 修改项 5：调整 `last_wrong_bonus`

**位置**: `recommendation.rs:192-196`

```
旧: 5.0
新: 3.0
```

**原因**: 系统已通过 `wrong_count`、`error_rate`、`forget_risk` 多方面体现了题目难度和遗忘程度，乘以 5 会造成重复加权。

---

### 修改项 6：调整 `randomness`

**位置**: `recommendation.rs:209-211`

```rust
// 旧: 0.9 + seed / 5000.0  --> 0.9 ~ 1.1
// 新: 0.95 + seed / 10000.0 --> 0.95 ~ 1.05
```

随机扰动适当减小即可，±5% 足够打破固定顺序。

### 修改项 8：新增 `reason` 推荐理由字段

**文件**: `recommendation.rs`，`RecommendedQuestion` 结构体

```rust
pub struct RecommendedQuestion {
    // ... 已有字段保持不变
    pub reason: Option<Vec<String>>,  // 新增
}
```

**理由生成规则与优先级排序**:

| 优先级 | 条件 | 理由文本 |
|--------|------|---------|
| 1 | `review_count == 0` | "新加入题目" |
| 2 | `forget_risk >= 1.0` 且 `due_at <= now` | "已到复习时间" |
| 3 | `forget_risk > 1.0` | "已超期N天" |
| 4 | `last_result == "wrong"` | "上次回答错误" |
| 5 | `error_rate >= 0.5` | "错误率XX%" |

**实现伪代码**:

```rust
fn generate_reason(
    forget_risk: f64,
    review_count: i64,
    last_result: &Option<String>,
    error_rate: Option<f64>,
    overdue_days: f64,
) -> Option<Vec<String>> {
    let mut reasons: Vec<(u8, String)> = Vec::new();

    if review_count == 0 {
        reasons.push((1, "新加入题目".to_string()));
    }
    if forget_risk >= 1.0 && overdue_days >= 0.0 {
        reasons.push((2, "已到复习时间".to_string()));
    }
    if overdue_days > 0.0 {
        reasons.push((3, format!("已超期{}天", overdue_days as i64)));
    }
    if let Some(r) = last_result {
        if r == "wrong" {
            reasons.push((4, "上次回答错误".to_string()));
        }
    }
    if let Some(rate) = error_rate {
        if rate >= 0.5 {
            reasons.push((5, format!("错误率{:.0}%", rate * 100.0)));
        }
    }

    if reasons.is_empty() {
        return None;
    }
    reasons.sort_by_key(|(p, _)| *p);
    Some(reasons.into_iter().map(|(_, t)| t).collect())
}
```

前端展示示例：

```
+----------------------+
| 新加入题目            |
| 已到复习时间          |
| 错误率80%             |
+----------------------+
```

### 修改项 10：双池选题 -- 暂缓

**原 V2 方案**: 每科 10 题，新题池 3 题 + 复习池 7 题
**V2.1 决策**: 暂缓实施
**原因**: 引入 `freshness_bonus`（最高 5x 加权）+ `forget_risk` 软上限后，新题曝光率大概率已显著改善。先观察实际推荐数据（新题占比、完成率、正确率），确认问题仍然存在后再考虑双池机制，避免过早增加系统复杂度。

## 四、`calculate_score()` 重写后伪代码

```rust
fn calculate_score(
    &self,
    question: &Question,
    now: Timestamp,
    review_count: i64,
    error_rate: Option<f64>,
) -> Result<(f64, f64, f64, f64, f64, f64), DbError> {
    // 1. forget_risk（软对数上限）
    let forget_risk = match (question.last_review_at, question.due_at) {
        (Some(last_review), Some(due)) => {
            let expected = ((due.as_i64() - last_review.as_i64()) as f64 / DAY_SECONDS as f64).max(1.0);
            let passed = ((now.as_i64() - last_review.as_i64()) as f64 / DAY_SECONDS as f64).max(0.0);
            (passed / expected + 1.0).log2()
        }
        _ => 0.0, // 从未复习
    };

    // 2. freshness_bonus（仅从未复习过的新题）
    let freshness_bonus = if review_count == 0 {
        let days = ((now.as_i64() - question.created_at.as_i64()) as f64 / DAY_SECONDS as f64).max(0.0);
        (5.0 - days * 0.25).max(1.0)
    } else {
        1.0
    };

    // 3. last_wrong_bonus
    let last_wrong_bonus = if let Some(ReviewResult::WRONG) = question.last_result { 3.0 } else { 1.0 };

    // 4. error_rate_bonus（带样本量平滑）
    let error_rate_bonus = match error_rate {
        Some(rate) => 1.0 + rate * ((review_count + 1) as f64).log2(),
        None => 1.0,
    };

    // 5. randomness
    let qid_i64: i64 = question.id.clone().into();
    let day_seed = (qid_i64 * 31 + now.as_i64() / DAY_SECONDS) % 1000;
    let randomness = 0.95 + (day_seed as f64 / 10000.0);

    // 6. 总分
    let score = (1.0 + forget_risk)
        * freshness_bonus
        * last_wrong_bonus
        * error_rate_bonus
        * randomness;

    Ok((score, forget_risk, freshness_bonus, last_wrong_bonus, error_rate_bonus, randomness))
}
```
